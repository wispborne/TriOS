# Harden Deep-Link Install

## Problem

The `starsector-mod://` deep-link install feature is implemented on the `wip-deep-links`
branch (`lib/trios/deep_link/`, wiring in `lib/main.dart` and settings), but a review
found bugs that break it during normal use, plus one adoption gap:

1. **Confirmation dialog crashes on first use.** The handler casts a `Notifier`'s `ref`
   to `WidgetRef` to open the confirm dialog. The cast throws at runtime, so the first
   link any user clicks (before "don't ask again" is set) fails to install anything.
2. **The launch URI is discarded, so reception is broken on Windows.** `main()` takes no
   `args`, so the URI the Windows runner forwards on argv is thrown away; reception relies
   solely on `appLinks.getInitialLink()`, which returns null on Windows. Confirmed live with
   a working launcher website: clicking a link (a) downloads nothing (cold path stores no
   URI) and (b) opens a new instance every time even when one is running (warm path skips
   forwarding). Compounding it, forwarding keys off mere existence of `running.lock`, which
   is also the crash-detection marker and is routinely left behind by a crash — so even once
   reception works, a stale lock would swallow a cold-start link.
3. **The popup has no reliable anchor.** There is no `GlobalKey<NavigatorState>` in the
   app; the handler grabs an arbitrary element to host its dialogs, so they may show in
   the wrong place or not at all.
4. **A `.version` file's download URL is trusted blindly and hidden.** The resolved
   `directDownloadURL` is downloaded without an http/https check and is never shown to the
   user, who only sees the mod name — so the approved mod and the actual byte source can
   differ with no visibility.
5. **"Already installed" detection is weak.** It only matches on `masterVersionFile`
   equality, which many mods lack, so it usually fails and re-downloads mods you have.
6. **Dead code.** `SingleInstanceManager.forwardDeepLinkIfSecondary()` is never called;
   `main.dart` reimplements forwarding inline.
7. **No way for authors to create links.** Authors would have to hand-encode
   `starsector-mod://install?mod=...` URLs, which nobody will do.
8. **Multiple links in quick succession are mishandled.** The handler's rate
   limiter is time-only (ignores URI identity), so clicking a second, *different*
   link within ~2s silently drops it. The IPC uses a single overwrite-only
   `pending_deeplink` file, so concurrent warm clicks lose all but the last. And
   two cold-start clicks can both become primary before either writes the lock,
   spawning two windows. There's no batching, so links that do get through each
   pop a separate confirmation dialog.

## Proposed Solution

Fix bugs 1–6 in the existing WIP code with surgical changes, and add an author-facing
"copy install link" action (7).

- **1 + 3**: Add a `navigatorKey` `GlobalKey<NavigatorState>` to the app's `MaterialApp`
  and have the handler use it for all dialogs. Change the confirm dialog to accept a `Ref`
  (or just the data it needs) instead of `WidgetRef`, so a `Notifier` can call it.
- **2**: Receive the URI from argv (`void main(List<String> args)`) on Windows/Linux — the
  native runner already forwards it — and keep app_links for macOS. Give the lock file a
  liveness signal (PID + alive check) so forwarding only happens when another instance is
  genuinely alive; a stale lock is treated as "not running" and the link is handled by this
  (cold-start) instance.
- **4**: Re-validate the resolved `directDownloadURL` is http/https (reject otherwise) and
  show the actual download domain in the confirmation dialog.
- **5**: Match installed mods by mod id as well, keeping the version comparison so updates
  are still detected.
- **6**: Route `main.dart` through `SingleInstanceManager` so the (now liveness-aware)
  forwarding logic lives in one place.
- **7**: Add a "Copy install link" action (mod context menu) that builds a correct
  `starsector-mod://install?mod=<encoded>` link and copies it to the clipboard.
- **8**: Queue incoming links instead of time-gating them: dedupe by URI identity
  (not a blanket time window), append each forwarded link to a `pending_deeplinks/`
  directory (one file per link, no overwrite), coalesce a burst into a single
  confirmation dialog that links clicked later merge into live, and claim the lock
  atomically on cold start so two clicks can't both become primary.

## Forum transport limitation (known, documented)

A forum test confirmed that a raw custom-scheme link **fails when clicked from a forum
post**: the forum opens it in a new window, and browsers refuse to launch a custom-protocol
handler from a fresh new-window navigation. The same URL works when pasted into the address
bar. So the link generated by (7) works for address-bar paste, Discord, and a future
landing page, but **not** for forum-post clicks. The robust long-term fix is an https
landing page that redirects to the scheme — recorded as a future option, not built here.

## Scope

- Bug fixes 1–6 in `lib/trios/deep_link/` and minimal wiring in `lib/main.dart` / the app
  root widget.
- New "copy install link" action (7) for mods.

## Non-goals

- Building or hosting the https landing-page web service.
- `nxm://` / NexusMods integration.
- Changing the `starsector-mod` scheme name.
- Reworking the download/install pipeline (`downloadAndInstallMod` is reused as-is).
- Adding automated tests for the protocol/lifecycle paths (verified manually).
